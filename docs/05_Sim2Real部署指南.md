## 核心挑战

Sim2Real（仿真到真实）的核心挑战是**仿真与现实之间的差距 (Reality Gap)**。这不是一个单一问题，而是多种小误差的累积叠加：

| 差距类型 | 具体表现 |
|---------|---------|
| 动力学差距 | 质量、惯性、摩擦、电机模型不精确 |
| 接触差距 | 接触力学、粘滑转换、柔性变形 |
| 传感器差距 | 延迟、噪声、偏置、采样率 |
| 执行器差距 | 力矩限制、热降额、齿轮间隙 |
| 环境差距 | 光照、地面材质、温度影响 |

## 1. 策略导出

### 1.1 从 Isaac Lab 导出模型

训练完成后，策略模型保存在 `logs/rsl_rl/<run_name>/` 目录下。

```python
"""export_policy.py - 导出策略为 ONNX 和 JIT 格式"""

import torch

# 方法 1：导出 TorchScript JIT 模型
policy = torch.jit.load("logs/rsl_rl/<run_name>/model_1500.pt")
policy.eval()

# 测试推理
dummy_obs = torch.randn(1, obs_dim)  # obs_dim = 你的观测维度
with torch.no_grad():
    action = policy(dummy_obs)
print(f"Action shape: {action.shape}")

# 方法 2：导出 ONNX（推荐用于边缘设备部署）
torch.onnx.export(
    policy,
    dummy_obs,
    "exported_policies/my_humanoid_policy.onnx",
    input_names=["observation"],
    output_names=["action"],
    dynamic_axes={"observation": {0: "batch"}, "action": {0: "batch"}},
)
print("ONNX model exported.")
```

训练时的观测归一化参数必须同时导出：

## 2. 部署架构

### 2.1 实机控制循环

```
┌─────────────────────────────────────────────────┐
│                 控制循环 (50~100 Hz)              │
│                                                   │
│  传感器读取 → 状态估计 → 观测构建 → 策略推理      │
│     ↑           │          │          │           │
│     │           │          │          ↓           │
│     │           │          │      动作后处理       │
│     │           │          │          │           │
│     │           │          │          ↓           │
│     └───────────┴──────────┴───── 电机控制 ◄──────┘
│                                                   │
└─────────────────────────────────────────────────┘
```

### 2.2 各环节职责

#### 传感器读取
- IMU：四元数姿态 + 角速度 + 线加速度
- 关节编码器：各关节位置 + 速度
- 可选：力/力矩传感器（脚底）

#### 状态估计
- 基座线速度估计（IMU 积分或外部定位）
- 基座角速度（直接从 IMU 读取）
- 重力投影计算

#### 观测构建
- 将传感器数据组装成与训练时完全一致的观测向量
- 应用归一化（使用训练时导出的 mean/var）
- 拼接上一步动作和速度命令

#### 策略推理
- ONNX Runtime 或 TorchScript 推理
- 输入：归一化后的观测向量
- 输出：各关节位置目标偏移量

#### 动作后处理
- `target_pos = default_joint_pos + action * action_scale`
- 可选：低通滤波平滑动作
- 可选：关节限位裁剪

#### 电机控制
- 将目标位置发送给各关节 PD 控制器
- PD 增益需与仿真中配置一致

## 3. 关键注意事项

### 3.1 动力学匹配（最重要）

这是 Sim2Real 成功与否的第一要素。

**必须做的事：**

- [ ] 用电子秤称量每个部件的实际质量
- [ ] 用 CAD 软件计算精确惯性矩阵
- [ ] 实测地面摩擦系数（橡胶/木地板/瓷砖差异很大）
- [ ] 测量电机力矩-速度曲线
- [ ] 测量控制回路延迟（从传感器读取到电机响应）
- [ ] 将上述参数更新到仿真 URDF/USD 中

**系统辨识方法：**

```
1. 录制实机数据（关节位置、速度、力矩）
2. 在仿真中重放相同的力矩序列
3. 对比仿真与实际的关节轨迹
4. 调整仿真参数直到匹配
5. 重复上述过程
```

### 3.2 Domain Randomization 参数指南

在训练阶段就加入 DR，使策略对参数变化具有鲁棒性：

| 随机化项 | 范围 | 说明 |
|---------|------|------|
| 各 link 质量 | +/- 10~30% | 对平衡影响大 |
| 地面摩擦 | 0.4 ~ 1.5 | 必须涵盖实际地面 |
| 关节 PD 刚度 | +/- 15~25% | 模拟电机特性差异 |
| 关节 PD 阻尼 | +/- 15~25% | 同上 |
| IMU 噪声 | std=0.02~0.1 | 高斯噪声 |
| 关节位置噪声 | std=0.01~0.03 rad | 编码器精度 |
| 动作延迟 | 0~2 control steps | 通信 + 计算延迟 |
| 外部推力 | 0~80 N | 随机方向推力 |
| 推力间隔 | 5~15 s | 不要太频繁 |
| 初始姿态扰动 | +/- 0.1 rad | 初始不完美站立 |
| 地形倾斜 | +/- 5° | 不完美平坦地面 |

### 3.3 观测空间对齐

**仿真与实机观测必须一一对应：**

```
仿真观测                    实机对应
─────────────              ──────────────
base_lin_vel (3)    ←→    IMU积分/外部定位估计
base_ang_vel (3)    ←→    IMU 陀螺仪直接读取
projected_gravity (3) ←→  IMU 姿态四元数 → 重力投影
velocity_commands (3) ←→  遥控器/上层规划输入
joint_pos (N)       ←→    关节编码器 - 零位偏移
joint_vel (N)       ←→    关节编码器差分/速度传感器
last_action (N)     ←→    上一步发出的动作
```

**关键检查项：**

- [ ] 坐标系一致（IMU 的 x/y/z 方向是否与仿真定义一致）
- [ ] 关节零位一致（编码器零位 = URDF 中的 0 位置）
- [ ] 关节正方向一致（正角度方向是否匹配）
- [ ] 单位一致（rad vs deg, m/s vs mm/s）
- [ ] 采样频率一致（control_dt 是否匹配）

### 3.4 控制频率匹配

| 参数 | 仿真值 | 实机值 | 要求 |
|------|--------|--------|------|
| 控制频率 | 50 Hz | 必须 50 Hz | 必须一致 |
| 物理频率 | 200 Hz | N/A（物理是真实的） | 仿真侧参数 |
| 推理耗时 | ~0ms（GPU 快） | <5ms（边缘设备） | 必须能在 control_dt 内完成 |

## 4. 常见问题与解决方案

### 4.1 "醉酒步态"（Drunken Robot Syndrome）

**症状**：机器人小碎步、犹豫、不敢大步迈出

**原因**：
- 仿真中的接触力学与现实不同
- 策略在仿真中学到的平衡策略在现实中不够用

**解决**：
1. 增大 Domain Randomization 范围（特别是摩擦、质量）
2. 增加外部推力扰动训练
3. 降低 action_rate 惩罚（允许更大的动作变化）
4. 考虑使用 PolySim 多仿真器训练

### 4.2 翻倒 / 不稳定

**症状**：机器人运行数秒后翻倒

**原因**：
- 惯性参数不匹配（特别是质心位置）
- 摩擦系数估计不准
- 控制延迟未被正确建模

**解决**：
1. 重新做系统辨识，校准惯性参数
2. 实测地面摩擦系数
3. 在训练中加入更大的延迟随机化
4. 先从低速命令开始（0.2 m/s），逐步提速

### 4.3 关节抖动 / 噪声

**症状**：关节快速来回震荡，有嗡嗡声

**原因**：
- action_scale 过大
- PD 增益不匹配
- 策略输出频率超过电机带宽

**解决**：
1. 减小 action_scale
2. 在动作输出端加低通滤波器
3. 增大训练中的 action_rate 惩罚
4. 检查并匹配 PD 增益

### 4.4 方向漂移

**症状**：机器人走着走着偏向一侧

**原因**：
- IMU 偏置未校准
- 左右腿电机特性不对称
- URDF 中左右不完全对称

**解决**：
1. IMU 上电时做零偏校准
2. 分别校准左右腿电机
3. 检查 URDF 对称性
4. 使用外部定位修正（Motion Capture / SLAM）

## 5. 实机部署分阶段流程

### 阶段 1：仿真 Zero-Shot 测试

用更精确的物理参数重建仿真环境，关闭所有 DR，测试策略表现。

```bash
# 用精确参数做 zero-shot 测试
./isaaclab.sh -p scripts/reinforcement_learning/rsl_rl/play.py \
  --task MyHumanoid-Velocity-Flat-v0 \
  --num_envs 1
```

### 阶段 2：悬挂测试

将机器人用绳索悬挂离地：
- [ ] 运行策略，观察各关节是否按预期运动
- [ ] 检查关节运动方向是否正确
- [ ] 检查关节幅度是否在安全范围内
- [ ] 确认无异常抖动或突变

### 阶段 3：扶持行走

一人扶住机器人躯干，给定低速前进命令：
- [ ] 观察步态是否合理
- [ ] 检查脚是否能正常抬起和落地
- [ ] 感受机器人的力度是否正常（不应太用力推）
- [ ] 记录问题点

### 阶段 4：低速自主行走

在安全区域（有防护垫）：
- [ ] 给定 0.2 m/s 前进命令
- [ ] 准备好随时接住机器人
- [ ] 观察稳定性，记录摔倒模式
- [ ] 如果稳定，逐步增加速度

### 阶段 5：正常速度行走

- [ ] 逐步增加到目标速度（如 0.5~1.0 m/s）
- [ ] 测试转弯功能
- [ ] 测试从静止到行走的切换
- [ ] 测试外部推力的恢复能力

### 阶段 6：地形适应

- [ ] 从完美平坦地面开始
- [ ] 轻微不平整地面
- [ ] 薄地毯/不同材质地面
- [ ] 小坡度斜面

## 6. 硬件端推理平台

### 6.1 NVIDIA Jetson Orin（推荐）

- 支持 TensorRT 加速，推理延迟 < 1ms
- 功耗 15~60W，适合机器人板载
- 支持 CUDA、cuDNN
- 价格约 $500~$2000

```python
# Jetson 上使用 TensorRT 推理
import tensorrt as trt
# 先将 ONNX 模型转为 TensorRT engine
# trtexec --onnx=policy.onnx --saveEngine=policy.trt
```

### 6.2 轻量级方案

```python
# 使用 ONNX Runtime（适用于各种平台）
import onnxruntime as ort

session = ort.InferenceSession("policy.onnx")
obs_np = observation.numpy().reshape(1, -1)
action = session.run(None, {"observation": obs_np})[0]
```

### 6.3 推理延迟要求

| 控制频率 | 可用推理时间 | 推荐平台 |
|---------|------------|---------|
| 50 Hz | < 20ms | 任何平台均可 |
| 100 Hz | < 10ms | Jetson Orin / x86 |
| 200 Hz | < 5ms | Jetson Orin + TensorRT |

## 7. 迭代改进循环

Sim2Real 不是一步到位的过程，需要持续迭代：

```
仿真训练 → 实机测试 → 发现问题 → 改进仿真/DR → 重新训练 → 再测试
   ↑                                                        │
   └────────────────────────────────────────────────────────┘
```

**每次迭代关注一个具体问题**，不要一次改太多东西。

记录每次实验的：
- 使用的策略 checkpoint
- 修改了什么参数
- 实机表现（最好录视频）
- 问题和改进方向
